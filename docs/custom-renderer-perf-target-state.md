# Final Target State: 自定义渲染器流式性能优化（增量 DOM 复用）

> 前置文档：`docs/custom-renderer-target-state.md`（Phase 1：自定义渲染器与 RenderBlock IR）。
> 本文档定义 Phase 1 之后的渲染端性能专项：问题定位、终态设计、复杂度账目、变更范围与验收标准。

## 一、当前状态与问题

Phase 1 的增量渲染已保证 **committed 块只渲染一次、DOM 缓存、永不重渲染**，该机制工作正常。

问题全部集中于 **active 区域（流式尾部）**：当前实现对它采取"每帧全量重渲染"策略。一帧 = 一次 `renderOutputItem` 调用（由 producer 的 `replaceOutput` 流式更新驱动）。每帧对 active 区域执行：

1. 清空 active 容器（`innerHTML = ''`）；
2. micromark **全量解析**整个已累积的 active markdown；
3. 对其中所有 `pre > code` 调用 lowlight/highlight.js **全量高亮**；
4. **重建全部 DOM**（含高亮产生的大量 span 子树）。

当单个 content / reasoning 块较长且包含多个大 fenced 代码块时，帧成本随累积长度线性增长，流式全程总成本平方增长，造成严重卡顿。

## 二、瓶颈定位（按昂贵程度排序）

1. **highlight.js 重复高亮（首要瓶颈）**：已闭合、内容从此不再变化的代码块，在后续每一帧都被重新高亮。开销 ≈ 帧数 × 已累积代码总量。
2. **DOM 全量重建**：innerHTML 重解析 + 高亮 span 子树重建 + 浏览器重排重绘。
3. **micromark 全量解析**：O(n)/帧，但其吞吐比 highlight.js 高一至两个数量级，聊天输出量级下不是主因。
4. **（独立瓶颈，本方案不处理）传输侧 O(n²)**：`agentRunner` 每帧 `JSON.stringify` 全量 RenderData（committed + active）并跨进程传输。留待 Phase 2 与 SSE 结构化事件改造合并处理。

## 三、可利用的关键不变量

1. **追加式输入（append-only）**：LLM 流式输出只增长、不回改。
2. **已闭合 fenced 代码块的内容从此不变**，其高亮结果可永久复用。
3. **每帧至多一个代码块处于增长状态**：未闭合的围栏会吞掉后续一切内容，因此当前帧解析出的代码块中只有最后一个可能仍在增长，此前的必然已闭合。这支撑 4.3 的成本有界性论证（且无需任何自建围栏扫描器）。
4. **高亮是确定性函数**：相同（语言, 文本）输入必然产生相同输出，因此未高亮内容指纹相同 ⟺ 高亮结果相同。

## 四、终态设计

**全部优化下沉为渲染端（`src/notebook/renderer.ts`）内部事务。IR、producer、serializer、SSE 零变更。**

### 4.1 核心机制：未高亮指纹对齐 + DOM 节点复用

active 区域从"每帧清空重建"改为简化版 keyed reconciliation。每帧管线（**顺序是硬性约束**）：

1. micromark 全量解析 active markdown，输出**未高亮** HTML，写入临时容器；
2. **顶层元素前缀对齐**：以未高亮内容指纹，比对临时容器与现有 active DOM 的顶层元素序列；指纹一致的前缀节点**直接保留**——连带其已完成的高亮、以及用户交互状态（如 `<details>` 开合）；
3. 从第一个分叉节点起，以新节点替换旧节点；若分叉节点是容器（列表、引用块等），先执行 **pre 打捞**（见 4.2）再替换；
4. **仅对新建节点中高亮成果缺失的 `pre` 执行高亮**（打捞失败或真正新建的）。

**硬性约束：对齐必须发生在高亮之前**（指纹取自 micromark 原始输出，而非高亮后的 outerHTML）。否则每帧为计算指纹就不得不先重高亮所有未变代码块，优化从内部失效。

**正确性论证**：

- 追加式流下，对齐命中率 ≈ "只有最后一个顶层元素在变化"；
- markdown 并非严格前缀稳定（GFM 表格在分隔行到达后才成立、setext 标题、链接引用定义可回溯改变前文渲染、列表松散性等）——在本方案下**自动安全，无需特判**：指纹比对的是**解析输出**而非对源文本的假设，输出变化则指纹变化，自然触发该节点重建；
- 附带修复一个现有 UX 缺陷：active reasoning / 流式 pendingTool 的 `<details>` 开合状态跨帧保留。由两个机制共同保证——文本不变的帧靠前缀复用保留节点；文本变化的帧（分叉替换）在替换前将旧节点 `<details>` 的 `open` 状态按 DOM 序继承到新节点。继承只发生在"同位置替换"路径，锁定/完成后的折叠语义（committed 追加新建壳）不受影响；指纹在继承修改前捕获，两机制互不干扰。

### 4.2 复用的统一：一切复用都是 DOM 节点保留，无独立高亮缓存

**不变量：任何已高亮的 `pre` 节点，只要其（语言, 文本）不变，就永远不会被重新高亮。**

顶层前缀对齐有一个结构性盲区：列表与引用块是**单个顶层元素**，代码块嵌在它们内部——容器内每增长一个 token，容器节点指纹即变化、整个子树重建（典型如"步骤列表 + 代码块"的教程式输出，列表内已闭合代码块会被反复重高亮）。覆盖该场景的机制不是缓存，而是对齐的延伸：

**分叉点 pre 打捞**：分叉的新旧容器子树之间，按（语言, 未高亮文本）逐比对双方的 `pre` 序列，文本一致的旧 `pre` 节点直接迁入新子树（连带已完成的高亮）；比对失败的（通常只有正在增长的尾部块）走正常高亮。列表松散性翻转（`<p>` 包裹出现/消失）等结构变化不影响 `pre` 自身结构，同样可打捞。

**锁定帧的跨容器 pre 打捞**：块锁定（L1/L2/L3）时，新 committed 块未高亮构建后、高亮前，从当前 active 树中**将被本帧 reconcile 淘汰的节点**（即前缀对齐分叉点之后的节点）按（语言, 文本）内容匹配打捞 `pre` 迁入（壳一律新建，仅迁移 `pre`，committed reasoning 的 collapsed 翻转不受影响）。存活的前缀节点不参与候选，避免从可见 DOM 中迁走节点。由此不变量对锁定帧同样成立：锁定不再伴随任何重复高亮。

**明确不引入独立的 `(语言, 文本) → 高亮 HTML` 字符串缓存**。其价值域被打捞完全覆盖，而打捞消除了缓存形态的全部管理负担：

- 无需 key 计算（长文本哈希）与容量管理（LRU）；
- 无需"已闭合才写入"的时机判定——打捞按文本比对，未闭合块文本逐帧变化，自然比对失败、正常高亮，不存在中间态垃圾问题；
- 高亮成果的唯一存储地就是 DOM 节点本身（无高亮 HTML 字符串副本）；对齐与打捞所需的未高亮指纹文本另存一份，空间与 active 内容量同阶、随节点离场即删，无历史累积。

### 4.3 未闭合尾部代码块策略

**维持每帧一次全量高亮**（与现状视觉行为一致，UX 零变化）。

- 这是必要成本而非浪费：highlight.js 是有状态的全量解析器（多行注释、跨行字符串等），不存在"只高亮新增部分"的增量模式；
- 成本有界：仅 O(当前块大小)，不再叠加全部历史代码块。

### 4.4 残留每帧成本（可接受，不进一步优化）

- micromark 全量解析 active：O(active 长度)，常数极小；committed 部分不参与解析（见 5.2）；
- 未高亮临时 DOM 构建：浏览器 innerHTML 解析是高度优化路径；
- 仅当未来 profiling 证明解析本身成为瓶颈，才考虑以"稳定前缀长度**提示**"形式做解析层优化（hint 而非语义分割：提示不精确只降低复用效率，不影响正确性）。**当前不做。**

## 五、IR 的定性：零变更

### 5.1 类型化块：跨进程语义契约（永久保留）

`RenderBlock`（content / reasoning / toolCall）的存在理由不是性能，而是语义：

- **toolCall 无法用 markdown 表达**：携带结构化 args、result、isStreaming、renderingConfig（代码块参数与语言推断），渲染端据此生成交互式 UI；
- reasoning 的 `collapsed` 是语义状态，不是文本；
- **非 DOM 消费方依赖块结构**（见 5.3 消费方图谱）。

### 5.2 committed/active 分割：角色重定位（保留）

建立 DOM 复用能力后，committed/active 分割与三级锁定（L1 轮结束 / L2 轮内 reasoning→content→tools / L3 工具完成）从"渲染性能机制"重新定位为：

1. **解析范围限定器**：committed 部分连 micromark 都不参与——每帧解析与临时构建成本为 O(active 长度) 而非 O(会话总长度)。这是 DOM 复用无法替代的独立节省；
2. **强不变性承诺**：契约级"不再变化"，渲染端对 committed 跳过一切对齐验证、append-only 追加。对齐算法的正确性负担被压缩在很小的 active 区域内；
3. **UX 状态机**：reasoning 在锁定时刻从展开切换为折叠；
4. **SSE 语义分段**：committed 追加对应"完成事件"，active 对应"流式 delta"——Phase 2 结构化事件改造的基础。

**结论：IR 粒度保持现状（整块 content、整块 reasoning）。块内部的细粒度复用是渲染端内部事务，不上升到契约层。**

### 5.3 IR 消费方图谱（全部零变更）

| 模块 | 角色 | 本次是否变更 |
|---|---|---|
| `notebook/renderTypes.ts` | IR 定义 | 否 |
| `notebook/renderer.ts` | 渲染端（进程内复制类型定义） | **是（唯一变更文件）** |
| `agent/uiRenderer.ts` | producer，三级锁定 | 否 |
| `agent/agentRunner.ts` | 传输（序列化 + replaceOutput） | 否 |
| `agent/toolExecutor.ts` | RenderBlock 回调 | 否 |
| `notebook/serializer.ts` | 会话持久化/恢复（buildInteractionRenderBlocks / renderBlocksToMarkdown） | 否 |
| `httpServer/chat.ts` | Headless/SSE 结构化消费 committed blocks | 否 |

## 六、时间/空间总账

| 项 | 每帧时间 | 额外空间 |
|---|---|---|
| **现状** | 全量 micromark + **全量代码块高亮** + 全量 DOM 重建 | O(1) |
| **终态** | micromark 全量解析（常数极小）+ 未高亮临时 DOM 构建 + **仅未闭合尾部块**高亮 O(k) + 仅分叉点后节点落 DOM | O(active 内容量)（指纹表保存未高亮 outerHTML 与 pre 文本各一份，与 active DOM 同阶；节点离场即删，**无历史累积**） |

- 已闭合代码块的每帧成本：**零**（节点保留，不解析产物比对之外的开销、不高亮、不重建）；
- 非 fenced 文本帧的高亮成本：**零**；
- 无中间态结构堆积，内存占用不随流式历史增长。

## 七、变更范围与规模

- **唯一变更文件**：`src/notebook/renderer.ts`（active 渲染路径重写为对齐复用式 + 分叉点 pre 打捞）；
- **无新增运行时依赖**：纯 DOM 实现，micromark/lowlight 管线不变；
- **规模**：中（一个简化版顶层元素 keyed reconciliation + 分叉容器内的 pre 序列比对迁移）；
- 渲染端测试/验证环境：notebook renderer webview。

## 八、验收标准与测试重点

1. **性能**：长 content + 多个大 fenced 代码块的流式场景下，单帧成本从"随累积长度线性增长"降为"近似常数（仅尾部块）"；主观卡顿消除；
2. **逐帧正确性**：增量渲染输出与同内容一次性全量渲染的输出一致。重点覆盖回溯场景：GFM 表格形成瞬间、setext 标题、链接引用定义后出现、列表松散性变化、嵌套围栏（列表/引用内代码块）；
3. **容器嵌套性能**：步骤列表/引用块内嵌多个已闭合代码块并持续流式增长时，已闭合 `pre` 不被重新高亮（打捞生效），帧成本不随已闭合代码总量增长；
4. **状态保留**：active reasoning `<details>` 的开合状态跨帧保留；
5. **回归**：committed 缩容清空路径（新对话轮 output 替换）行为不变；`disposeOutputItem` 后无 DOM 缓存泄漏；JSON 解析失败帧之后的成功帧必须恢复到可见 DOM 正常渲染（失败时作废旧 cache，恢复时重建干净容器，"无 cache ⟺ 宿主内容不被任何 cache 引用"）；
6. **空间**：流式全程内存占用平坦，无随帧数增长的结构堆积。

## 九、范围边界

**做：**
- 渲染端未高亮指纹对齐 + DOM 节点复用（active 渲染路径重写）；
- 分叉点 pre 打捞（容器嵌套场景的高亮免除）；
- active `<details>` 交互状态保留（附带修复）。

**不做：**
- IR / producer / serializer / httpServer(SSE) / toolExecutor 的任何变更；
- 传输侧 O(n²) 优化（归入 Phase 2，与 SSE 结构化事件改造合并设计）；
- 未闭合代码块的高亮节流或纯文本降级（维持每帧高亮，UX 零变化）；
- 解析层"稳定前缀"优化（仅当未来 profiling 证明 micromark 成为瓶颈，且只以 hint 形式引入）；
- 高亮引擎替换（TextMate 系支持行级增量 tokenization，是远期可选项，现阶段不引入）；
- `renderOutputItem` 的 abort signal 响应（已知限制：渲染为同步路径，优化后单次耗时已大幅下降，暂不引入异步取消）；
- lowlight 语言集 `all → common` 裁剪（低优先级独立事项，仅影响 bundle 体积与注册开销）。

## 十、决策记录

- 未闭合尾部代码块：维持每帧一次全量高亮（必要成本，UX 零变化优先）；
- 复用机制收敛为单一不变量"（语言, 文本）不变的 `pre` 节点永不重高亮"，由顶层对齐 + 分叉点打捞 + 锁定帧跨容器打捞实现；不引入任何形式的 `(语言, 文本) → HTML` 字符串缓存（其价值域被打捞覆盖，且打捞消除 key 计算、容量管理与写入时机判定的全部管理负担）；
- 对齐实现粒度：micromark 输出的 HTML 顶层元素指纹，零新增依赖；
- committed/active 分割与三级锁定保留，角色重定位为"范围限定器 + 强不变性承诺"；
- 传输侧 O(n²) 留待 Phase 2。
